## 1. 项目背景与用户痛点 (Background & Problem)

* **背景：** 随着各大高校和期刊部署严苛的 AI 检测工具（如 Turnitin, GPTZero），许多非母语研究者（ESL）因使用翻译软件或固定模版，即使是原创内容也常被误判为 AI 生成。同时，部分学生确实需要对 AI 辅助生成的内容进行“去 AI 化”润色。
* **痛点：**
1. 现有的检测工具缺乏解释性（只给分，不给理由）。
2. 市面上的降重工具（Paraphraser）容易改变学术原意（Semantic Drift）。
3. 单一的 Prompt 改写（方案 A）容易被更新的检测器识别，单一的规则替换（方案 B）会导致语句不通顺。



---

## 2. 核心功能模块 (Core Functional Modules)

系统分为三个主要阶段：**输入预处理 -> AIGC 诊断（检测） -> 混合引擎优化（降重）**。

### 2.1 模块一：智能预处理 (Smart Preprocessing)

**目标：** 保护学术严谨性，防止乱改专业术语。

* **F.1.1 文本清洗：** 去除参考文献列表（Bibliography）、图表标题，避免干扰检测。
* **F.1.2 术语锁定 (Term Protector)：**
* 用户可上传自定义“术语白名单”。
* 系统通过 NER (命名实体识别) 自动识别专有名词（如 "Deep Residual Learning", "ANOVA"），在后续流程中标记为 `<LOCKED>`，**严禁修改**。



### 2.2 模块二：多维 AIGC 诊断系统 (Analysis Engine)

**目标：** 实现你提出的“4维度分析”，提供可解释的检测报告。

| 分析维度 | 检测指标/算法逻辑 | 输出内容 |
| --- | --- | --- |
| **1. 用词分析** | **困惑度 (PPL):** 使用 GPT-2/LLaMA 计算。<br>

<br>**AI 指纹密度:** 匹配 "delve", "crucial", "realm" 等高频词库。<br>

<br>**词汇丰富度 (TTR):** 检测词汇重复率。 | 高亮显示“平庸词”；<br>

<br>给出 PPL 曲线图。 |
| **2. 语法分析** | **句法树深度:** 分析句子的分支复杂度（AI 偏向中等深度正态分布）。<br>

<br>**语法完美度:** 检测是否存在“过于完美的语法”（零错误率）。 | 标记“过于机械”的句子结构。 |
| **3. 结构分析** | **突发性 (Burstiness):** 计算句子长度的标准差。<br>

<br>**段落均质性:** 检测段落长度是否过于整齐。 | 给出“节奏评分”；<br>

<br>警告“像列表一样的段落”。 |
| **4. 逻辑/思路** | **语义连贯性:** 计算相邻句子的 Cosine Similarity。<br>

<br>**连接词依赖:** 统计 "However", "Therefore" 的使用频率。 | 标记强行逻辑连接的地方。 |

### 2.3 模块三：混合人源化引擎 (Hybrid Humanizer Engine)

**目标：** 结合方案 A 和 方案 B，降低 AIGC 分数。

#### 流程步骤：

1. **Stage A (重构 - The Brain):**
* **机制：** 调用大模型 API (GPT-4o / Claude 3.5)。
* **策略：** 使用少样本 (Few-shot) Prompt，要求模型模拟“资深审稿人”口吻，增加长短句变化，打乱原有的生成逻辑。


2. **Stage B (清洗 - The Surgeon):**
* **机制：** 基于 Python 规则 + BERT Masked LM。
* **动作 1 (禁词替换):** 扫描 Stage A 的输出，强制替换残留的 AI 指纹词（如将 "paramount" 替换为 "essential" 或重写短语）。
* **动作 2 (噪音注入):** 在语法合规处随机插入副词、插入语，打破 PPL 预测链。
* **动作 3 (句式微调):** 强制将部分“主谓宾”结构改为倒装句或强调句。



### 2.4 模块四：质量卫士 (Quality Gate)

**目标：** 防止改写后的文章“变味”。

* **F.4.1 语义一致性检查：** 计算 `Embedding_Similarity(原句, 改写句)`。如果相似度 < 0.85，自动回滚或重新生成。
* **F.4.2 可读性校验：** 计算 Flesch-Kincaid 分数。如果文章变得极度难读，提示用户警告。

## 3. 技术架构与算法规划 (Technical Architecture)

### 3.1 技术栈推荐

* **后端框架:** FastAPI (Python) - 高性能，适合 ML 部署。
* **NLP 核心库:**
* `spaCy`: 用于分词、句法树分析、实体识别。
* `Transformers (Hugging Face)`: 加载 BERT/RoBERTa 模型进行特征替换。
* `PyTorch`: 深度学习计算框架。


* **大模型接口:** OpenAI API 或 Anthropic API。
* **前端:** React.js 或 Streamlit (快速原型)。

### 3.2 算法流程图逻辑

```text
[用户输入] 
   ↓
[术语锁定模块 (NER)] ---> 输出带 <MASK> 的文本
   ↓
[检测模块] ---> 输出 4维报告 & AIGC Score
   ↓ (如果 Score > 阈值)
[Engine A: LLM重写] ---> Prompt: "High Burstiness, Varied Sentence Length"
   ↓
[Engine B: 规则清洗] 
   ├─> 替换 Stopwords (AI指纹)
   ├─> 注入 Syntax Noise
   └─> 还原 <MASK> 术语
   ↓
[质量卫士] ---> 语义相似度 > 0.85? --(No)--> 重新生成
   ↓ (Yes)
[输出最终文本]

```

## 4. UI/UX 交互设计 (Brief Design)

* **布局：** 左右分栏对比模式（Left: Original, Right: Humanized）。
* **交互细节：**
* **高亮系统：**
* 🔴 红色背景：高风险 AIGC 句子。
* 🟡 黄色下划线：AI 惯用词（指纹词）。
* 🟢 绿色背景：已优化且安全的句子。


* **仪表盘：** 顶部显示 4 个维度的雷达图。
* **控制台：** 用户可调节“改写力度”滑块（Low/Medium/Aggressive）。
* *Low:* 仅微调词汇（Engine B only）。
* *Aggressive:* 重写整个结构（Engine A + B）。

## 6. 风险与应对 (Risks)

1. **误杀专业词汇：**
* *应对：* 必须强制执行“术语锁定”逻辑，宁可 AIGC 分数高一点，也不能改错学术名词。


2. **API 成本过高：**
* *应对：* Engine A 是成本大头。对于长论文，采用“按需重写”策略（只重写红色的高风险段落），而不是全文重写。


3. **检测器对抗升级：**
* *应对：* Engine B 的词库需要定期更新（每月一次），根据最新的 AI 写作习惯维护黑名单。
